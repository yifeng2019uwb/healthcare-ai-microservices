# Healthcare Infrastructure

Database schema and Synthea synthetic data for the healthcare platform.

## What's Running

| Component | Details |
|-----------|---------|
| Database | Supabase PostgreSQL |
| Schema | One SQL file per table in `schema/sql/` |
| Test data | Synthea synthetic patient data (HIPAA-safe) |

## Directory Structure

```
healthcare-infra/
├── schema/
│   ├── run-schema.sh       # deploy DDL to Supabase
│   └── sql/                # one file per table
│       ├── users.sql
│       ├── organizations.sql
│       ├── patients.sql
│       ├── providers.sql
│       ├── encounters.sql
│       ├── conditions.sql
│       ├── allergies.sql
│       ├── audit_logs.sql
│       └── ...
└── synthea/
    ├── run-synthea.sh      # generate synthetic patient data
    └── synthea-with-dependencies.jar
```

## Deploy Schema

```bash
cd healthcare-infra/schema

# Deploy all tables (safe to re-run — IF NOT EXISTS guards)
DATABASE_URL="postgresql://postgres:<password>@db.<ref>.supabase.co:5432/postgres" \
  ./run-schema.sh

# Deploy a single table
DATABASE_URL="..." ./run-schema.sh encounters

# Using a .env file
echo 'DATABASE_URL=postgresql://...' > ../.env
./run-schema.sh
```

## Generate Synthea Data

```bash
cd healthcare-infra/synthea

# Generate data for n patients
./run-synthea.sh <n>

# Generate test-only CSV data for admin integration tests
./run-synthea.sh test-data <n>
```

## Schema Design Notes

- Each table has its own SQL file; files are idempotent (`CREATE TABLE IF NOT EXISTS`, `CREATE INDEX IF NOT EXISTS`)
- Tables are deployed in dependency order (users → organizations → patients → providers → encounters → ...)
- Indexes are explicit — no query should run without an index
- Audit logs are append-only (no DELETE — enforced at DAO layer)
- UUIDs are assigned by the application layer, not generated by the DB
