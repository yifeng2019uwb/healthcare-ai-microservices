# Healthcare Infrastructure

Database schema and Synthea synthetic data for the healthcare platform.

## What's Running

| Component | Details |
|-----------|---------|
| Database | Supabase PostgreSQL |
| Schema | One SQL file per table in `schema/sql/` |
| Test data | Synthea synthetic patient data (HIPAA-safe) |

## Deploy Schema

`DATABASE_URL` is auto-loaded from `docker/.env` if not set in the environment.

```bash
cd healthcare-infra/schema

# Deploy all tables (safe to re-run)
./run-schema.sh

# Deploy or migrate a single table
./run-schema.sh ai_analysis_results

# Explicit DATABASE_URL override
DATABASE_URL="postgresql://user:password@host:5432/dbname" ./run-schema.sh
```

## Adding a Column to an Existing Table

Schema files are idempotent — `CREATE TABLE IF NOT EXISTS` skips creation if the table already exists. To add a new column to a live table, append `ALTER TABLE ADD COLUMN IF NOT EXISTS` **before** any index that references the new column:

```sql
-- Migrations (idempotent — safe to re-run, must run before indexes that use these columns)
ALTER TABLE my_table
    ADD COLUMN IF NOT EXISTS new_column UUID REFERENCES other_table(id);

-- Index (references new_column — must come after the ALTER)
CREATE INDEX IF NOT EXISTS idx_my_table_new_column
    ON my_table (new_column);
```

Order matters: `ALTER TABLE` first, then the `CREATE INDEX` that uses it.

## Generate Synthea Data

```bash
cd healthcare-infra/synthea

# Generate data for n patients into the default output dir
./run-synthea.sh <n>

# Generate test-only CSV data for admin integration tests
./run-synthea.sh test-data <n>
```

## Schema Design Notes

- Each table has its own SQL file; files are fully idempotent (`IF NOT EXISTS` throughout)
- Tables are deployed in FK dependency order (see `run-schema.sh` `ALL_TABLES` array)
- Column additions use `ALTER TABLE ADD COLUMN IF NOT EXISTS` in the same file — no separate migration files
- Indexes are explicit — no query should run without an index
- `audit_logs` and `ai_analysis_results` are append-only (no UPDATE/DELETE — enforced at DAO layer)
- UUIDs are assigned by the application layer, not generated by the DB
